# Phase 9.1: Library Foundation - Context

**Gathered:** 2026-05-06
**Status:** Ready for planning

<domain>
## Phase Boundary

`pkg/tldt` is a stable, externally-importable Go library covering all core capabilities (summarize, detect injection, sanitize Unicode, fetch URLs, full Pipeline). The CLI routes these core operations through `pkg/tldt` rather than calling `internal/` packages directly, proving the library is production-ready for embedding in external Go programs.

This phase is a pure refactor + verification — no new capabilities, only making the existing library load-bearing.

</domain>

<decisions>
## Implementation Decisions

### Integration Test Coverage
- **D-01:** Comprehensive test coverage at pkg/tldt layer
- Add edge case tests at public API layer: empty input, invalid algorithms, injection detection, Unicode sanitization, SSRF blocking
- Validate the public API contract, not just that functions are callable
- Trust internal package tests for exhaustive correctness, but replicate key scenarios at public API boundary
- Include tests for: all algorithms (lexrank, textrank, graph, ensemble), sanitize+detect+summarize Pipeline, Fetch with error cases

### Fetch Function API Design
- **D-02:** Add structured FetchResult type with error context (hybrid approach)
- pkg/tldt.Fetch should return `(FetchResult, error)` where FetchResult contains:
  - `Text string` — extracted article text (primary use case)
  - `StatusCode int` — HTTP status for consumers to handle 404 vs 500
  - `ContentType string` — to verify HTML was processed
  - `FinalURL string` — URL after redirects
- Question raised: core pkg/tldt is text processing (summarize/detect/sanitize), not HTTP — Fetch is a convenience utility
- Keep Fetch simple for summarization use case; metadata fields for middleware consumers who need them
- Add sentinel error types for common cases: ErrHTTPError (non-2xx), ErrNonHTML

### Error Wrapping Strategy
- **D-03:** Wrap all errors with context using `fmt.Errorf("tldt.FunctionName: %w", err)`
- Every exported function wraps errors from internal packages with its name
- Example: `tldt.Summarize: invalid algorithm "foo"`
- Tradeoff accepted: Callers cannot `errors.Is(err, internal.ErrSomething)` directly
- If specific error checking needed, define pkg/tldt sentinel errors (ErrInvalidAlgorithm, ErrSSRFBlocked already re-exported)

### main.go Refactor Strategy
- **D-04:** Big-bang refactor — replace all 4 internal imports in one commit
- Replace internal/summarizer, internal/detector, internal/sanitizer, internal/fetcher imports with pkg/tldt
- 417 lines of main.go updated atomically
- 344 existing tests serve as safety net
- Single commit: "refactor: route CLI through pkg/tldt library surface"

### Scope Clarification
- **D-05:** internal/config, internal/formatter, internal/installer remain direct imports in main.go
- pkg/tldt wraps only: summarizer, detector, sanitizer, fetcher (the 4 core text-processing packages)
- Config, formatter, installer are CLI-specific infrastructure, not part of embeddable library API

</decisions>

<specifics>
## Specific Ideas

- pkg/tldt is the "authoritative API surface" — external Go programs should only need to import this one package
- Integration test should prove: a program importing only `github.com/gleicon/tldt/pkg/tldt` can perform full sanitize→detect→summarize pipeline
- The "pipe-safe" design (stats to stderr, summary to stdout) stays in cmd/tldt/main.go — not pkg/tldt concern
- Phase 9.1 success criteria (from ROADMAP.md):
  1. `go test ./pkg/tldt/...` passes with integration test exercising full round-trip
  2. `grep` for internal/summarizer, internal/detector, internal/sanitizer, internal/fetcher in main.go returns no matches
  3. All 344+ tests pass after refactor

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- pkg/tldt/tldt.go: Public API skeleton exists with Summarize, Detect, Sanitize, Fetch, Pipeline
- pkg/tldt/tldt_test.go: 184 lines of tests covering basic functionality — extend per Decision D-01
- Internal packages: summarizer.New(), detector.Analyze(), sanitizer.SanitizeAll(), fetcher.Fetch() — wrapped by pkg/tldt

### Established Patterns
- Error wrapping: Already used in pkg/tldt (`fmt.Errorf("tldt.Summarize: %w", err)`)
- Sentinel errors: ErrSSRFBlocked, ErrRedirectLimit already re-exported via `var ErrX = internal.ErrX`
- Option structs: SummarizeOptions, DetectOptions, FetchOptions, PipelineOptions pattern established
- Result structs: Result, DetectResult, SanitizeReport, PipelineResult pattern established

### Integration Points
- cmd/tldt/main.go lines using internal packages:
  - `summarizer.New(algorithm)` → route through `tldt.Summarize()`
  - `detector.Analyze(text)` → route through `tldt.Detect()` or `tldt.Pipeline()`
  - `sanitizer.SanitizeAll(text)` → route through `tldt.Sanitize()` or `tldt.Pipeline()`
  - `fetcher.Fetch(url, timeout, maxBytes)` → route through `tldt.Fetch()`
  - `detector.DetectPII(text)` — currently CLI-only, stays internal until Phase 10 adds public API
  - `detector.DefaultOutlierThreshold` → move to pkg/tldt or keep internal (used in flag default)

### Refactor Notes
- `injectionThreshold` flag uses `detector.DefaultOutlierThreshold` — either:
  - Move constant to pkg/tldt as `tldt.DefaultOutlierThreshold`
  - Hardcode default in main.go (0.85)
- `--detect-pii` and `--sanitize-pii` flags call internal/detector directly — these are Phase 9 features (already done), not Phase 10 (LIB-01/02 public API)
- Wait: detect-pii/sanitize-pii are already CLI features from Phase 9. Phase 10 makes them available via pkg/tldt public API. Phase 9.1 just routes existing internal calls through pkg/tldt where they already exist (Summarize, Detect, Sanitize, Fetch, Pipeline).

</code_context>
